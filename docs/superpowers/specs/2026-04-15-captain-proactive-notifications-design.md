# Captain Proactive Notifications — Design Spec

**Date:** 2026-04-15
**Status:** Approved
**Author:** Mike + Claude

## Overview

A background worker running on the existing Callova Railway server that monitors Mike's world and pushes notifications to the Captain mobile app. Three tiers: urgent (sound), normal (standard), silent (badge only).

## Notification Tiers

| Tier | Trigger | Sound | Expo Priority |
|------|---------|-------|---------------|
| Urgent | Booking cancellation, weather warning with bookings, system issues | Sound + vibrate | high |
| Normal | New booking, morning briefing (7am PT), evening wrap-up (8pm PT) | Standard | default |
| Silent | Business metrics, learner insights | Badge only | low |

## Background Worker

Runs as cron jobs in the existing `src/crons.js` on Railway. No new services.

### Every 5 Minutes — Booking Sync + Diff

1. Fetch current bookings from `callova_bookings` table for business ID
2. Compare against `captain_world_state` (source: `notifications`, category: `last_booking_snapshot`)
3. If new booking detected: push normal notification with booking details
4. If cancellation detected: push urgent notification
5. Update snapshot in world state

### Every 15 Minutes — Weather Check

1. Search `"weather Kelowna BC today"` via Brave Search API
2. Parse results for wind speed, precipitation chance, temperature
3. Check if bookings exist within next 24 hours
4. If wind >40km/h OR rain >80% AND upcoming bookings: push urgent weather warning
5. Store latest weather in world state (source: `weather`, category: `current`)

### 7:00am PT — Morning Briefing

1. Sync Callova bookings and weather
2. Generate briefing via Claude Haiku using: today's bookings, weather, world state, identity
3. Push as normal notification with full briefing text
4. Store in conversation history

### 8:00pm PT — Evening Wrap-up

1. Compare today's bookings vs what was expected (completed, cancelled, new)
2. Preview tomorrow's schedule
3. Generate wrap-up via Claude Haiku
4. Push as normal notification

## Architecture

### New File: `src/captain/notifications.js`

Responsibilities:
- `diffBookings(current, previous)` — returns `{ added: [], cancelled: [] }`
- `checkWeatherAlert(weatherData, upcomingBookings)` — returns alert or null
- `sendTieredNotification(tier, title, body, data)` — wraps push.js with Expo priority/sound fields
- `buildBookingNotification(booking, type)` — templates for new/cancelled booking alerts

### Changes to Existing Files

**`src/captain/push.js`:**
- Update `sendPushNotification()` to accept `priority` and `sound` parameters
- Map tiers: urgent → `{ priority: 'high', sound: 'default' }`, normal → `{ priority: 'default', sound: 'default' }`, silent → `{ priority: 'low', sound: null }`

**`src/captain/briefing.js`:**
- Add weather data to briefing context (pull from world state)
- Add booking summary to briefing context

**`src/crons.js`:**
- Add four new cron jobs:
  - `captainBookingSync` — every 5 minutes
  - `captainWeatherCheck` — every 15 minutes
  - `captainMorningBriefing` — daily at 7:00am America/Vancouver
  - `captainEveningWrapup` — daily at 8:00pm America/Vancouver

### Mobile App

No changes needed. Push notifications already registered on app launch via `expo-notifications`. Tapping a notification opens the Captain app (default Expo behavior).

## Notification Content Templates

**Morning briefing (generated by Claude Haiku):**
> Morning Mike. 2 parasail bookings today — 11am (party of 3) and 2pm (party of 2). Weather: 22C, sunny, winds 15km/h. No issues.

**New booking:**
> New booking: {activity}, {date} at {time}, party of {party_size}.

**Cancellation:**
> Cancellation: {date} {time} {activity} (party of {party_size}) was just cancelled.

**Weather warning:**
> Heads up: winds forecasted at {wind}km/h {tomorrow/today}. You have a booking at {time} — might need to reschedule.

**Evening wrap-up (generated by Claude Haiku):**
> Today: {completed} parasail trips, {new_bookings} new bookings. Tomorrow: {tomorrow_count} booking(s) at {times}. Weather looks {condition}.

## Data Flow

```
Cron tick
  → fetch current state (bookings / weather)
  → diff against last snapshot in captain_world_state
  → if change detected → determine tier
  → build notification (template for alerts, Claude Haiku for briefings)
  → send via Expo Push API with tier-appropriate priority
  → update captain_world_state with new snapshot
```

## Weather Parsing

Query Brave Search for `"weather Kelowna BC today"`. Extract from search result descriptions:
- Temperature (look for pattern like `\d+°C`)
- Wind speed (look for `\d+ km/h`)
- Precipitation (look for `\d+%` near rain/precipitation)
- General condition (sunny, cloudy, rain, etc.)

Store parsed weather in `captain_world_state` (source: `weather`, category: `current`).

## Environment Variables

All existing — no new env vars needed:
- `CAPTAIN_BUSINESS_ID` — which business to monitor (already set to 1)
- `BRAVE_SEARCH_API_KEY` — for weather queries (already set)
- Push token — already registered by mobile app

## Success Criteria

1. Mike receives a morning briefing push at 7am PT with bookings + weather
2. Mike receives an evening wrap-up at 8pm PT
3. New FareHarbor bookings trigger a push within 5 minutes
4. Cancellations trigger an urgent push within 5 minutes
5. Bad weather with upcoming bookings triggers an urgent alert within 15 minutes
6. Tapping any notification opens Captain app
